Understanding "Fetching Live Data"

An architectural breakdown of the caching layer and its rate-limiting behaviors

The Log Context

During the pipeline run, you will see output like this:

Found 692 candidates below 200-DMA. Checking Market Caps...
📡 Fetching live data for 1301.T...
📡 Fetching live data for 1379.T...
📡 Fetching live data for 1414.T...
📡 Fetching live data for 1419.T...

This message means the local cache proxy system bypassed local storage and fetched data directly from the Yahoo Finance API.

How the Caching Proxy Works

To avoid hitting yfinance limits, our scripts route requests through a custom caching layer in scripts/yf_proxy.py. When a ticker is requested:

  1. File Check: It looks for a cached file named [TICKER].json inside the daily folder scratch/cache/[YYYYMMDD]/ (or the debug folder scratch/cache/debug/ if running with YF_DEBUG=1).
  2. Validation Check: It validates the file schema. For example, it checks if "quarterly_financials" exists.
  3. Deciding Action:
    • If the file exists and is valid, the script loads it instantly from disk (silently).
    • If the file does not exist (or lacks key data), it outputs 📡 Fetching live data for [TICKER]..., queries the API, and caches the fresh response.

Historical Batch Download Timeout & Pace

During the initial screening phase, we fetch 5 years of daily price history for the entire market (e.g., 1,552 Japan tickers) in chunks of 20 tickers at a time:

  • Expected Speed: Downloading and parsing 5 years of history for 20 tickers typically takes ~4 to 6 seconds per chunk under normal Yahoo Finance API conditions. For 1,552 tickers (78 chunks), a full live scan from scratch takes ~6 to 8 minutes.
  • Strict Timeout: To prevent the entire script from hanging indefinitely if a single ticker or connection stalls, we pass a strict timeout=10 seconds to the batch download method.
df_chunk, is_cached = proxy_download(
    chunk, period="5y", interval="1d", group_by='ticker',
    progress=False, threads=True, timeout=10
)

If any chunk times out, the script catches the exception, logs it, and immediately proceeds to the next chunk, preserving pipeline progress.

Parallel Querying & Thread-Safe Rate Limiting

Resolution Optimized Concurrency & Safe Pacing To make the pipeline fast yet highly stable, the Market Cap Verification Loop is parallelized using a ThreadPoolExecutor. To protect our IP from getting blacklisted, we enforce a thread-safe sliding-window rate limiter inside the caching proxy.

1. Concurrency: Parallel Thread Pool (scripts/screener.py)

Instead of single-threaded sequential querying, candidates are processed concurrently in 10 worker threads, using a lock to ensure thread safety when writing results:

with ThreadPoolExecutor(max_workers=10) as executor:
    futures = {executor.submit(check_candidate, c): c for c in candidates}
    for future in as_completed(futures):
        progress.advance(task)

2. Safe Pacing: Sliding-Window Rate Limiter (scripts/yf_proxy.py)

A lock-protected rate limiter intercepts all live API calls, ensuring they never exceed **100 requests per 60 seconds**. Threads calling the API will auto-sleep if they hit the rate limit cap, pacing requests safely:

class RateLimiter:
    def wait_if_needed(self):
        with self.lock:
            now = time.time()
            self.requests = [t for t in self.requests if now - t < self.period]
            if len(self.requests) >= self.max_requests:
                sleep_time = self.period - (now - self.requests[0])
                time.sleep(sleep_time)
            self.requests.append(now)

Best Practices

  • Safe Concurrency: The worker threads are I/O bound. The 10 concurrent connections drastically reduce overall wait times without overloading CPU resources.
  • Caching Synergy: Cached files bypass both the rate-limiter and the thread pool's network overhead entirely, loading in fractions of a millisecond.
  • Debug Environment: You can still run with YF_DEBUG=1 to use the offline cache for testing logic changes instantly.
