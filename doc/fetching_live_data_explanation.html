Understanding "Fetching Live Data"

An architectural breakdown of the caching layer and its rate-limiting behaviors

The Log Context

During the pipeline run, you will see output like this:

Found 692 candidates below 200-DMA. Checking Market Caps...
📡 Fetching live data for 1301.T...
📡 Fetching live data for 1379.T...
📡 Fetching live data for 1414.T...
📡 Fetching live data for 1419.T...

This message means the local cache proxy system bypassed local storage and fetched data directly from the Yahoo Finance API.

How the Caching Proxy Works

To avoid hitting yfinance limits, our scripts route requests through a custom caching layer in scripts/yf_proxy.py. When a ticker is requested:

  1. File Check: It looks for a cached file named [TICKER].json inside the daily folder scratch/cache/[YYYYMMDD]/ (or the debug folder scratch/cache/debug/ if running with YF_DEBUG=1).
  2. Validation Check: It validates the file schema. For example, it checks if "quarterly_financials" exists.
  3. Deciding Action:
    • If the file exists and is valid, the script loads it instantly from disk (silently).
    • If the file does not exist (or lacks key data), it outputs 📡 Fetching live data for [TICKER]..., queries the API, and caches the fresh response.

The Rate-Limiting Blind Spot

Vulnerability Sequential Requests Without Delay Unlike the initial screening phase (which downloads chunks of 20 tickers with a 1-second pause), the Market Cap Verification Loop queries each candidate individually and sequentially with zero sleep intervals.

Here is the loop code in scripts/screener.py:

for c in candidates:
    try:
        t = ProxyTicker(c['symbol']) # Queries cache or runs live request
        info = t.info
        mcap = info.get('marketCap', 0)
    ...
    progress.advance(task) # NO delay sleep is present here!

If you have 692 candidates below the 200-DMA, and 100 of them are missing from the cache, the script will trigger **100 rapid-fire live requests** to the yfinance API. This high frequency is highly likely to trigger rate limits, returning empty responses or blocking your IP.

Best Practices to Avoid Blocks

  • Maintain Caches: Running the pipeline daily keeps the cache warm, reducing the number of missing tickers.
  • Use Debug Mode: When testing script logic or report layout modifications, run with DEBUG=1 make pipeline-jpx so it forces local reads from the frozen cache directory.
  • Safe Fallbacks: If needed, we can implement a 0.5-second cooldown in `screener.py` that only triggers if a live fetch was executed during the market cap check.
