🐛 Bug Report: Empty JP/TW Results in make all

Why dma_200_screen_result_link_jp.html shows 0 stocks in report/last/

Diagnosed: 2026-06-27

Summary

When running make all, the JP (and TW) screener reports in report/last/ show 0 companies found, even though the dated folder (report/20260627_report/) contains valid results. Two independent bugs contribute to this failure.

Bug #1: Yahoo Finance Rate Limiting

⚡ Root Cause — Silent API Failures

make all processes ~4,000 tickers sequentially across three markets. By the time JP screening starts, Yahoo Finance is rate-limiting requests. Failed batch downloads are silently swallowed, causing 0 stocks to pass the filter.

Pipeline Load

PhaseMarketTickersAPI Batches (20/batch)Status
1US (S&P 500 + NASDAQ-100)~600~30✅ OK
2TW (TWSE + TPEx)~1,800~90⚠️ Partially rate-limited
3JP (TSE Prime)~1,500~75❌ Heavily rate-limited

Silent Exception in screener.py

The batch download loop catches all exceptions without logging, so rate-limit errors are invisible:

# screener.py — screen_stocks() — lines 132-144
for i in range(0, len(tickers), chunk_size):
    chunk = tickers[i:i+chunk_size]
    try:
        df_chunk, is_cached = proxy_download(
            chunk, period="5y", interval="1d",
            group_by='ticker', progress=False,
            threads=True, timeout=10
        )
        if not df_chunk.empty:
            data_frames.append(df_chunk)
    except Exception:      # ← ALL errors silently swallowed
        is_cached = False
        pass              # ← rate-limit errors vanish here
    progress.advance(task, advance=len(chunk))

Why the 2nd run succeeds

The proxy cache layer (yf_proxy.py) saves successful downloads as .pkl files. On the 2nd run:

  • Previously successful batches → loaded instantly from disk cache (no API call)
  • Previously failed batches → re-downloaded successfully (rate-limit window has passed)
# yf_proxy.py — proxy_download() — lines 345-351
if os.path.exists(cache_path):
    print(f"📦 Loading cached batch data ({len(tickers)} tickers)...")
    df = pd.read_pickle(cache_path)
    return adjust_multi_splits(df), True  # ← returns cached, no API hit

Bug #2: Makefile update-last Only Runs Once

⚡ Root Cause — Shared .PHONY Prerequisite

Even when the 2nd run produces valid JP data, report/last/ is never updated because Make only executes the shared update-last target once across all three pipelines.

Buggy Makefile (before fix)

update-last was listed as a prerequisite of each pipeline. Since it's a shared .PHONY target, Make considers it "already done" after the first execution:

# Makefile — BEFORE fix
.PHONY: pipeline-us
pipeline-us: screen-us reports-us links-us update-last

.PHONY: pipeline-tw
pipeline-tw: screen-tw reports-tw links-tw update-last

.PHONY: pipeline-jp
pipeline-jp: screen-jp reports-jp links-jp update-last

What Make actually executed

pipeline-us:

screen-us→ reports-us→ links-us→ update-last ✅

Syncs report/last/ — but JP/TW files are empty (from 1st run)

pipeline-tw:

screen-tw→ reports-tw→ links-tw→ update-last ❌ SKIPPED

pipeline-jp:

screen-jp→ reports-jp→ links-jp→ update-last ❌ SKIPPED

Result: report/last/ still has the empty JP/TW files from pipeline-us's sync.

Evidence: make -n all dry run (before fix)

./.venv/bin/python3 scripts/add_report_links.py --market tw

# ← no cp here either!

Fix Applied

✅ Solution — Move update-last into recipe body

By calling @$(MAKE) update-last in the recipe instead of listing it as a prerequisite, Make executes it independently for each pipeline.

Diff

# Makefile — AFTER fix
 .PHONY: pipeline-us
- pipeline-us: screen-us reports-us links-us update-last
+ pipeline-us: screen-us reports-us links-us
+ 	@$(MAKE) update-last
  	@echo "✅  Full pipeline complete for US market."

 .PHONY: pipeline-tw
- pipeline-tw: screen-tw reports-tw links-tw update-last
+ pipeline-tw: screen-tw reports-tw links-tw
+ 	@$(MAKE) update-last
  	@echo "✅  Full pipeline complete for TW market."

 .PHONY: pipeline-jp
- pipeline-jp: screen-jp reports-jp links-jp update-last
+ pipeline-jp: screen-jp reports-jp links-jp
+ 	@$(MAKE) update-last
  	@echo "✅  Full pipeline complete for JP market."

Remaining Consideration

⚠️ Rate Limiting Is Expected Behavior

The 1st run of each day may still produce incomplete JP/TW results due to Yahoo Finance rate limits (~4,000 tickers exceeds the ~50 req/min budget). This is by design — the cache system accumulates successful downloads across runs, and subsequent make all invocations fill in the gaps. The critical fix ensures that when data IS available, it always reaches report/last/.
